## Full re-review: Shop-Quote `quote-wizard.tsx` — live testing readiness

I've read the complete file end-to-end. Here's what I found with fresh eyes, judged against the single standard that matters: **can a machinist in a CNC shop open this, key in a job, and fire off a quote in under 3 minutes?**

---

### What is genuinely working ✅

The core architecture is now correct — single screen, card-per-part, live price in the header of every card, Basic/Advanced toggle, sticky summary panel on the right, collapsible Terms section below. The `calculateCosts` function is solid and does proper margin-on-cost (not markup) arithmetic. Customer defaults auto-applying on select is a great timesaver. The scan assist panel logic is complete and well-structured. The success/email screen is clean and practical.

---

### Issues that will hurt live testing — fix before deploying

**1. Dead wizard code is still live and running** *(will cause subtle bugs)*

`currentStep`, `STEPS`, `handleNext`, `handlePrev` are all still in the component. They're not in the rendered UI — but `handleFinalSubmit` still calls `setCurrentStep(3)` to trigger the success screen. This means the success screen is controlled by wizard step state, not a proper `saved` boolean. If a save fails (API error), `currentStep` still advances to 3 and shows "Quote Saved!" even though nothing was saved. **Fix**: replace `currentStep === 3` guard with a `const [quoteSubmitted, setQuoteSubmitted] = useState(false)` flag, set it only after `onSubmit` completes successfully.

**2. TemplatePicker has no escape hatch** *(blocks quick quoting)*

On every new quote the picker appears with no way to skip it. A machinist who already knows the job type and just wants to type in numbers cannot dismiss it without clicking a template first. The description even still says *"preload sensible defaults for step 3"* — that's wizard language from a previous version and will confuse users. **Fix**: add a `<button onClick={() => setTemplateApplied(true)}>Skip →</button>` link and update the description to say "You can adjust everything afterwards."

**3. RFQ Received Date and Follow-up Date are in the primary info bar** *(wrong place for SME shop)*

These are CRM tracking fields. They sit in Row 2 of the info bar, taking up two of four columns, level with Lead Time and Delivery Terms. For a small shop doing quick quotes, these are admin fields — they should live inside the Terms/Notes collapsible section. Right now the info bar is too wide and these fields are confusing to a machinist who just wants to say "4 weeks, Ex Works." **Fix**: move `rfqReceivedDate` and `followUpDate` into the Follow-up subsection inside the collapsible, and replace their slots in Row 2 with Lead Time presets (currently taking the full col-span) displayed more compactly.

**4. `PartTabs` component is defined but never rendered** *(dead code, multi-part navigation broken)*

`const PartTabs = () => ...` is defined inside the component but never called anywhere in the JSX. When a user adds a second part, there is no tab strip to switch between parts — only the `activeLineItemIndex` state clicking on the card itself works. In a multi-monitor shop environment where cards are side-by-side (xl: grid-cols-2), this is fine. But on a laptop at 1280px, two cards stack vertically and the part you're not editing just sits there active-highlighted from the last click. The `PartTabs` component should either be rendered above the parts grid or removed entirely.

**5. `form.watch()` called per field inside a `fields.map` loop** *(performance issue, causes re-renders on every keystroke)*

Inside the parts loop: `const item = form.watch(`lineItems.${idx}`)` — this subscribes the entire parent component to every change on every line item simultaneously. With multiple parts open, every keystroke in any input re-renders the whole component. **Fix**: extract each part card into a separate `PartCard` sub-component that only watches its own index.

**6. The sticky summary panel's JSX is incomplete in the file** *(confirmed build error)*

The file cuts off mid-render at `const item = form.watch(\`lineItems` — the closing backtick, the rest of the map, the summary totals, the Save button, and the closing `</div></div></form>` tags are all missing. This is a **real truncation in the raw file itself** at ~34KB. The file on GitHub is likely being served truncated. You need to check the actual Replit file directly — if it's the same, the component will fail to compile. **This is your highest-priority check.**

**7. The `ArrowLeft` icon is imported but never used** *(minor, but causes a lint warning)*

`import { ..., ArrowLeft, ..., Save, ... }` — both `ArrowLeft` and `Save` are imported but not rendered anywhere. Remove them.

**8. `riskPercentage` defaults to 10 regardless of settings** *(logic bug in `newItemDefaults()`)*

```ts
riskPercentage: settings?.defaultMarginPercentage ? 10 : 10,
```
This ternary does nothing — both branches return 10. Should be the actual default risk from settings, e.g. `settings?.defaultRiskPercentage ?? 10`.
